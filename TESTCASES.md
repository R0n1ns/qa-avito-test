# Тест-кейсы для API объявлений (Avito QA Internship)

## 1. Создание объявления (POST /api/1/item)

### 1.1 Позитивные сценарии и контракты
* **1.1.1: Создание валидного объявления.**
  * *Входные данные:* sellerID (111111-999999), name (string), price (>0), statistics (integers).
  * *Ожидаемый результат:* Статус 200 OK. Ответ содержит объект с полями `id`, `sellerId`, `name`, `price`, `statistics`, `createdAt`.
* **1.1.2: Проверка схемы (контракта) ответа.**
  * *Ожидаемый результат:*
    - `id`: string (UUID).
    - `sellerId`: integer.
    - `name`: string.
    - `price`: integer.
    - `statistics`: объект с integer полями (`likes`, `viewCount`, `contacts`).
    - `createdAt`: string (валидный формат ISO 8601).
* **1.1.3: Граничные значения числовых полей.**
  * *Входные данные:* `price = 0` (бесплатное), `price = 1`. `sellerId = 111111` и `999999`. Статистика = 0.
  * *Ожидаемый результат:* 200 OK, данные сохранены корректно.
* **1.1.4: Валидация даты создания (createdAt).**
  * *Ожидаемый результат:* Поле `createdAt` соответствует ISO 8601, значение времени в прошлом или равно текущему (не "из будущего").

### 1.2 Негативные сценарии (Параметры и Заголовки)
* **1.2.1: Отсутствие обязательных полей.**
  * *Входные данные:* JSON без `name`, или без `sellerId`, или без `price`.
  * *Ожидаемый результат:* 400 Bad Request.
* **1.2.2: Отсутствие блока или полей статистики.**
  * *Входные данные:* JSON без объекта `statistics` или без поля `likes` внутри него.
  * *Ожидаемый результат:* 400 Bad Request.
* **1.2.3: Проверка заголовка Content-Type.**
  * *Входные данные:* POST запрос с `Content-Type: text/plain` или без заголовка.
  * *Ожидаемый результат:* 400 Bad Request или 415 Unsupported Media Type.
* **1.2.4: Невалидный JSON (Malformed JSON).**
  * *Входные данные:* Пропущенная кавычка или запятая в теле запроса.
  * *Ожидаемый результат:* 400 Bad Request.

### 1.3 Негативные сценарии (Валидация значений)
* **1.3.1: Пустое имя товара.** (`name: ""`).
* **1.3.2: Превышение лимита символов.** (`name` > 256/1000 символов).
* **1.3.3: sellerId вне диапазона.** (111110 или 1000000).
* **1.3.4: Неверный тип данных.** (`price` как string, `likes` как boolean).
* **1.3.5: Отрицательные значения.** (`price = -1`, `contacts = -10`).
* **1.3.6: Переполнение целого числа (Integer Overflow).** (`price` > 2^63-1).
* **1.3.7: Инъекции (XSS/SQL).** Попытка передать `<script>alert(1)</script>` в поле `name`.
  * *Ожидаемый результат для всех:* 400 Bad Request (или 200 OK с экранированием для инъекций).

### 1.4 Корнер-кейсы и системные проверки
* **1.4.1: Идемпотентность и дубли.** При отправке двух идентичных запросов должны создаться два объекта с разными `id`.
* **1.4.2: Уникальность ID.** Создать 10 объявлений подряд — все `id` в ответах должны быть уникальными.
* **1.4.3: Игнорирование лишних полей.**
  * *Входные данные:* JSON + поле `"admin": true`. Лишнее поле не должно сохраняться.
* **1.4.4: Конкурентность (Race Condition).**
  * *Шаги:* 5 одновременных потоков создают объявления.
  * *Ожидаемый результат:* Все 5 созданы успешно, конфликтов ID нет.

## 2. Получение по ID (GET /api/1/item/:id)

### 2.1 Позитивные сценарии
* **2.1.1: Получение существующего объявления.**
  * *Ожидаемый результат:* 200 OK. Ответ содержит **массив объектов** (может быть пустым или содержать записи). Структура каждого элемента: `id`, `sellerId`, `name`, `price`, `statistics`, `createdAt`.
* **2.1.2: Сквозная консистентность данных (POST == GET).**
  * *Шаги:* Создать объект -> Получить его по ID.
  * *Ожидаемый результат:* В GET-ответе (массиве) присутствует элемент с тем же `id`, что и в POST-ответе. Все поля (цена, имя, статистика, sellerId) в этом элементе в точности равны значениям из POST-запроса.

### 2.2 Негативные сценарии
* **2.2.1: Несуществующий ID.** -> 404 Not Found.
* **2.2.2: Неверный формат ID.** (Спецсимволы или слишком длинная строка). -> 400/404.
* **2.2.3: Пустой ID.** (`GET /api/1/item/`). -> 404.

## 3. Получение всех объявлений продавца (GET /api/1/:sellerID/item)

### 3.1 Позитивные сценарии
* **3.1.1: Продавец с несколькими объявлениями.** Должен вернуться массив всех созданных объектов.
* **3.1.2: Консистентность списка.** Проверка, что данные объектов внутри массива не "побились" и соответствуют исходным.

### 3.2 Негативные сценарии
* **3.2.1: Продавец без объявлений.** -> 200 OK и пустой массив `[]`.
* **3.2.2: Невалидный sellerID.** (Буквы вместо цифр). -> 400 Bad Request.

---

## 4. Получение статистики (GET /api/1/statistic/:id)

### 4.1 Позитивные сценарии
* **4.1.1: Проверка контракта статистики.** Ответ должен содержать **массив объектов** статистики. Каждый элемент содержит только `likes`, `viewCount`, `contacts` (integers).
* **4.1.2: Консистентность статистики.** Проверка, что хотя бы один элемент из массива GET-ответа содержит значения `viewCount`, `likes`, `contacts`, совпадающие с тем, что мы передавали в POST-запросе при создании объявления.

### 4.2 Негативные сценарии
* **4.2.1: Статистика для несуществующего ID.** -> 404 Not Found.

## 5. End-to-End (E2E) сценарии

### E2E-1: Полный жизненный цикл объявления
1. **POST /api/1/item**: Создаем объявление "iPhone 15" с sellerID = 555555, price = 50000, statistics = {likes: 10, viewCount: 100, contacts: 5}. Получаем ID из ответа (например, "uuid-123").
2. **GET /api/1/item/:id**: Запрашиваем /api/1/item/uuid-123. Проверяем, что в массиве ответа присутствует объект с этим ID и поля name, price, sellerId, statistics совпадают с отправленными.
3. **GET /api/1/statistic/:id**: Запрашиваем /api/1/statistic/uuid-123. Проверяем, что в массиве присутствует объект со статистикой likes=10, viewCount=100, contacts=5.
4. **GET /api/1/:sellerID/item**: Запрашиваем /api/1/555555/item. Проверяем, что в массиве присутствует объявление с ID "uuid-123" и именем "iPhone 15".
* *Ожидаемый результат:* Полная целостность данных на всех этапах, объявление найдено во всех ответах.

### E2E-2: Массовая проверка продавца
1. Создаем 3 разных товара для одного sellerID = 666666 (item1, item2, item3). Сохраняем их ID.
2. Запрашиваем список через `/api/1/666666/item`.
* *Ожидаемый результат:* Массив содержит минимум 3 элемента (может содержать больше, если другие тестировщики создавали товары для этого sellerId). Все 3 созданных ID присутствуют в массиве.

## 6. Примечяния API

### Структура ответов
- **GET /api/1/item/:id** возвращает массив объектов, не одиночный объект
- **GET /api/1/statistic/:id** возвращает массив объектов статистики, не одиночный объект
- **GET /api/1/:sellerID/item** возвращает массив объектов продавца
- Все массивы могут быть пустыми, если данных нет

### Особенности тестирования на общем стенде
- Тесты должны работать в условиях, когда данные могут создаваться другими пользователями
- Нельзя полагаться на фиксированное количество объектов - проверяйте наличие созданных вами объектов в массивах
- При проверке консистентности используйте поиск по ID или уникальным комбинациям полей
